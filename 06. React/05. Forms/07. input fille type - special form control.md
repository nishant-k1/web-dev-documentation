# input file type Form Control is a special case — it's neither fully uncontrolled nor uncontrolled

You can't control it (browser blocks value=) — that part is uncontrolled
But you still use onChange + state to store the File object — that part is controlled
This is not a React design choice — it is a browser-enforced security rule.

For reading and submitting, onChange is enough, But for reseting we can't do `setFormData({ file: null });`.
`useRef` is needed if you want to reset the file input.

**Since you cannot do this:**

```jsx
setFormData({ file: null }); // ❌ visually does nothing
```

Without the ref, your state resets to null but the input still shows the filename — they go out of sync visually.

**You must use a ref:**

```jsx
const fileRef = useRef(null);

const clearFile = () => {
  fileRef.current.value = "";
  setFormData((prev) => ({ ...prev, file: null }));
};

<input type="file" ref={fileRef} onChange={handleChange} />;
```

✅ All standard form controls can be reset via component state  
 ❌ The only exception is <input type="file">

```js
<input type="file" name="file" ref={fileRef} onChange={handleChange} />
```

useRef is only needed if you want to reset the file input. For reading and submitting, onChange is enough.

## Understanding File Input Values

1. value
   The `value` of a file input is a fake path string — not the actual file.

   ```js
   e.target.value; // "C:\fakepath\photo.jpg"  ← useless, always fakepath
   e.target.files; // FileList  ← this is what you actually use
   ```

   The browser intentionally hides the real file path for security — you can never know where the file lives on the user's computer.
   That's why e.target.value is useless for file inputs and e.target.files is the only way to get the actual file.

   ```sql
   file input   →  e.target.value    = "C:\fakepath\photo.jpg"  ← fake, ignore this
               →  e.target.files    = FileList          ← use this instead
               →  e.target.files[0] = File object       ← the actual file
   ```

2. files[0] or files
   files[0] object is just metadata — not the actual bytes:

   ```js
   file.name; // "photo.jpg"
   file.size; // 204800  (bytes count)
   file.type; // "image/jpeg"
   file.lastModified; // 1714123456789  (timestamp)
   ```

   The actual content (the bytes) is not read yet. File is just a reference to the file sitting on the user's disk.

## Reading File Content (Optional)

- FileReader
- Usually not needed for uploads

To actually read the content you need FileReader:

```js
const reader = new FileReader();
reader.readAsDataURL(file); // reads as base64 string
reader.onload = (e) => {
  console.log(e.target.result); // now you have the actual content
};
```

## API Integration: Sending File to Backend

Before sending data, you must decide:

JSON (application/json)
OR FormData (multipart/form-data)

👉 This is NOT React-specific

Real-world perspective

Think of it like:

JSON → sending a document
FormData → sending a parcel with items inside

If you include even one physical item (file),
you must use the parcel system

HTTP request body must be one format only
Either:
application/json
OR multipart/form-data

Not both together

**Multipart request (FormData)**
Supports both text + binary
That’s why we switch formats

You are not “losing JSON”
You are switching the transport format of the entire request

If all fields are text → use JSON
If even one field is a file → use FormData

text only form → JSON.stringify + Content-Type: application/json
form with files → new FormData() + NO Content-Type header

JSON → you set Content-Type manually
FormData → never touch Content-Type, let the browser set it

- Use FormData
- JSON cannot handle File objects
- Browser handles binary reading

```js
const data = new FormData();
data.append("name", formData.name);
data.append("email", formData.email);
data.append("dp", formData.dp); // browser reads the bytes when sending
fetch("/api/upload", { method: "POST", body: data });
```

But doing this for large number of fields manually is not practical

You should do this .

```js
const data = new FormData();

Object.entries(formData).forEach(([key, value]) => {
  if (value) data.append(key, value);
});
```

**FormData** is:

1. Flat (no nesting)
2. String-based (except File)
3. Manually constructed

**Key edge cases**:

1. Empty values
2. File absence
3. Multiple files
4. Arrays & nested objects
5. Incorrect headers
6. File validation (type & size)
7. Large uploads
8. Backend mismatches
9. Debugging FormData

# Events and Form Controls

Note: we attach and remove event handlers NOT events itself, events are implicit to elements provided by the browser based on the element type.

Elements don't have fixed "event listening capabilities" built into them. Instead, the browser triggers specific types of events based on user interactions and element behavior, and those events can then propagate through the DOM. We can listen to and handle those events using React props like onClick, onChange or methods like addEventListener.

The browser determines which event to trigger based on a combination of user interaction and the element's behavior (type and state)

Note: **Event = interaction + element behavior**

## User interactions

1. types
2. clicks
3. hovers
4. focuses
5. presses keys
6. drags
7. scrolls

## Event propagation (event bubbling and event capturing)

1. Comes in picture in case of hierachia/nested elements
2. event propagation (bubbling and capturing) takes place always, but only those element that have handler for that event will respond
3. event always originates at the target element
4. When an event triggers on the target element both bubbling phase and capturing phase always takes place.

### Event flow

It has three phases in the following order

1. Capturing phase (top -> down, document -> target element)
2. Target phase (actual element)
3. Bubbling phase (down -> up, target element -> document)

The capturing phase always happens BEFORE the bubbling phase,
Both phases happen, exactly once, in a fixed order

### Event Bubbling

Event starts at the target element and moves upward through its ancestors

```text
target child element -> parent -> grandparent -> document
```

**handler running order**
child handler runs first -> parent handler next and grand parent handler next and so on

### Event Capturing

Event starts from the document and travels downward to the target
Example Flow:

```text
document -> grandparent -> parent -> target child element
```

**handler running order**
grand parent handler -> parent handler runs first -> child handler next and so on

### e.stopPropagation()

Stops the event from moving further
If called during capturing -> stops going down
If called during bubbling -> stops going up
It does not cancel the event itself
It does not stop default behavior

## Event Delegation

Attaching a single event listener to a parent element to handle events for its children

Instead of adding handlers to every child

- Add one handler on parent
- Use event.target to figure out which child triggered it

## Common Event Types you should know

**`By default bubbling events are used everywhere in React and capturing events are rarely used`**

1. **Common Events for Form-Controls (All are bubble phase events)**
   1. onChange
   2. onFocus
   3. onBlur
   4. onClick

1. **Common Events for non form-controls**
   1. onSubmit
   2. onReset

## Less Common Event Types you should know

1. **Less Common Events for form-controls (Majority are capture phase events)**
   1. onInput and onInputCapture
   2. onChangeCapture
   3. onFocusCapture
   4. onBlurCapture

2. **Less Common Events for both form controls and non form-controls**
   1. onKeyDown and onKeyDownCapture
   2. onKeyUp and onKeyUpCapture
   3. onClickCapture
   4. onSubmitCapture

## When to use and when not to use e.preventDefault()

1. e.preventDefault() stops the browser's default behavior for an event

   Every HTML element has some built-in behavior  
   preventDefault() = "Don't do that default thing"

2. Only events that have a default browser action need preventDefault()

   If there's no default action -> Nothing to prevent -> Calling it is useless

3. These events need preventDefault()
   1. **onSubmit** (MOST IMPORTANT): Default behavior is Page reload + form submission to server
   2. **onClick** (SOMETIMES)
      - "button has submit type inside form": Default behavior is Page reload + form submission to server
      - "Anchor tag `<a>`": Default behavior is Navigates to URL
   3. **onKeyDown** (SOMETIMES)

4. These events doesn't need preventDefault(): There's no separate default action happening after the event
   1. **onChange**
   2. **onBlur**
   3. **onFocus**

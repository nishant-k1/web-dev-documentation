# Events and Form Controls

Note: we attach and remove event handlers NOT events itself, events are implicit to elements provided by the browser based on the element type.

Elements don't have fixed "event listening capabilities" built into them. Instead, the browser triggers specific types of events based on user interactions and element behavior, and those events can then propagate through the DOM. We can listen to and handle those events using React props like onClick, onChange or methods like addEventListener.

The browser determines which event to trigger based on a combination of user interaction and the element's behavior (type and state)

**Event = interaction + element behavior**

## User interactions

- types
- clicks
- hovers
- focuses
- presses keys
- drags
- scrolls

In React, all form handling revolves around:

1. Events (onChange, onSubmit)
2. Event object (e)
3. Target (e.target)
4. State (controlled components)

If you master what e.target gives for each control, you're done

## Event propagation (event bubbling and event capturing)

1. Comes in picture in case of hierachia/nested elements
2. event propagation (bubbling and capturing) takes place always, but only those element that have handler for that event will respond
3. event always originates at the target element
4. When an event triggers on the target element both bubbling phase and capturing phase always takes place.

### Event flow

It has three phases in the following order

1. Capturing phase (top -> down, document -> target element)
2. Target phase (actual element)
3. Bubbling phase (down -> up, target element -> document)

The capturing phase always happens BEFORE the bubbling phase,
Both phases happen, exactly once, in a fixed order

### Event Bubbling

Event starts at the target element and moves upward through its ancestors

```text
target child element -> parent -> grandparent -> document
```

**handler running order**
child handler runs first -> parent handler next and grand parent handler next and so on

### Event Capturing

Event starts from the document and travels downward to the target
Example Flow:

```text
document -> grandparent -> parent -> target child element
```

**handler running order**
grand parent handler -> parent handler runs first -> child handler next and so on

### e.stopPropagation()

Stops the event from moving further
If called during capturing -> stops going down
If called during bubbling -> stops going up
It does not cancel the event itself
It does not stop default behavior

## Event Delegation

Attaching a single event listener to a parent element to handle events for its children

Instead of adding handlers to every child

- Add one handler on parent
- Use event.target to figure out which child triggered it

## Common Event Types you should know

**`By default bubbling events are used everywhere in React and capturing events are rarely used`**

**1. Common Events for Form-Controls (All are bubble phase events)**

- onChange
- onFocus
- onBlur
- onClick

**2. Common Events for non form-controls**

1. onSubmit
2. onReset

1. onKeyDown and onKeyDownCapture

## Events + Target Mapping for form controls

What to read from target

1. **all input types (except checkbox and file type), textarea and single option select (dropdown) - only single option select**
   - event: `onChange`
   - what to read from target: `e.target.value`
2. **checkbox input single option select**
   - event: `onChange`
   - what to read from target: `e.target.checked`

3. **checkbox input multiple option select**
   - event: `onChange`
   - what to read from target: `manual array build`

4. **file input**
   - event: `onChange`
   - what to read from target: `e.target.files`

5. **multiple options select (dropdown)**
   - event: `onChange`
   - what to read from target: `Array.from(e.target.selectedOptions, o => o.value)`

## Events + Target mapping for non form controls

1. button and submit type button

- event: `onClick`
- what to read from target: `Nothing to read but if button has type submit then must do e.preventDefault() to prevent browsers default behaviour`
  Usually nothing to read, but you may read e.target, e.currentTarget, or dataset

1. form - onSubmit

- event: `onSubmit`
- what to read from target: `Nothing to read but do preventDefault() to prevent browsers default behaviour`

bubbling events are default and capture events are rarely used.

## One Handler for all inputs (Interview Gold)

Prefer using one event handler function for all the form controls

```jsx
const [formData, setFormData] = useState({
  name: "",
  email: "",
  password: "",
  agree: false,
  skills: [],
  gender: "",
  file: null,
  hobbies: [],
});
```

```jsx
<input
  type="text"
  name="name"
  value={formData.name}
  onChange={handleChange}
/>

<input
  type="email"
  name="email"
  value={formData.email}
  onChange={handleChange}
/>

<input
  type="password"
  name="password"
  value={formData.password}
  onChange={handleChange}
/>

<input
  type="checkbox"
  name="agree"
  checked={formData.agree}
  onChange={handleChange}
/>

<label>
  <input
    type="checkbox"
    name="skills"
    value="js"
    checked={formData.skills.includes("js")}
    onChange={handleChange}
  />
  JavaScript
</label>

<label>
  <input
    type="checkbox"
    name="skills"
    value="react"
    checked={formData.skills.includes("react")}
    onChange={handleChange}
  />
  React
</label>

<input
  type="radio"
  name="gender"
  value="male"
  checked={formData.gender === "male"}
  onChange={handleChange}
/>

<input
  type="radio"
  name="gender"
  value="female"
  checked={formData.gender === "female"}
  onChange={handleChange}
/>

<input
  type="file"
  name="file"
  onChange={handleChange}
/>

<select
  multiple
  name="hobbies"
  value={formData.hobbies}
  onChange={handleChange}
>
  <option value="1">Option 1</option>
  <option value="2">Option 2</option>
  <option value="3">Option 3</option>
</select>
```

```js
const handleChange = (e) => {
  const { type, name, value, checked, files, multiple, selectedOptions } =
    e.target;

  setFormData((prev) => {
    let finalValue;

    if (type === "checkbox") {
      // Handle checkbox groups (arrays)
      if (Array.isArray(prev[name])) {
        finalValue = checked
          ? [...prev[name], value]
          : prev[name].filter((v) => v !== value);
      } else {
        finalValue = checked;
      }
    } else if (type === "file") {
      finalValue = files[0]; // single file
    } else if (multiple) {
      finalValue = Array.from(selectedOptions, (o) => o.value);
    } else {
      finalValue = value;
    }

    return {
      ...prev,
      [name]: finalValue,
    };
  });
};
```

## Common Mistakes (VERY IMPORTANT)

1. Using value for checkbox input type
2. Using value for file input type
3. Using value for multi options select and using selectedOptions for single option select
4. Not setting value prop for the form control in controlled component
5. Forgetting preventDefault which causes page reload

## When to use and when not to use e.preventDefault()

1. e.preventDefault() stops the browser's default behavior for an event

   Every HTML element has some built-in behavior  
   preventDefault() = "Don't do that default thing"

2. Only events that have a default browser action need preventDefault()

   If there's no default action -> Nothing to prevent -> Calling it is useless

3. These events need preventDefault()
   1. **onSubmit** (MOST IMPORTANT): Default behavior is Page reload + form submission to server
   2. **onClick** (SOMETIMES)
      - "button has submit type inside form": Default behavior is Page reload + form submission to server
      - "Anchor tag `<a>`": Default behavior is Navigates to URL
   3. **onKeyDown** (SOMETIMES)

4. These events doesn't need preventDefault(): There's no separate default action happening after the event
   1. **onChange**
   2. **onBlur**
   3. **onFocus**

# What Happens When `setState` Runs in a Controlled component

Example:

```javascript
const handleChange = (event) => {
  setInputQuery(event.target.value);
};
```

When this runs:

```
input event
↓
setInputQuery()
↓
React schedules update
↓
component function runs again
```

So yes:

```
The entire component function re-executes.
```

---

# Does the Whole Component Re-render?

Yes.

React **calls the entire component function again**.

Example:

```javascript
function App() {
  const [inputQuery, setInputQuery] = useState("");

  const filteredData = users.filter(...);

  return (
    <>
      <input />
      <table />
    </>
  );
}
```

When state changes:

```
App() runs again
filteredData recalculates
JSX gets recreated
```

---

# Important: Re-render ≠ DOM Update

React does **NOT replace the entire DOM**.

Instead it performs:

```
Virtual DOM diffing
```

Process:

```
Old Virtual DOM
vs
New Virtual DOM
↓
React finds differences
↓
Only changed DOM nodes update
```

Example:

If only the input value changed:

```
<input value="a" />
```

becomes

```
<input value="ab" />
```

React only updates that DOM property.

The table DOM may remain untouched.

---

# Why This Is Not a Performance Problem

Running a component function is **very cheap**.

Typical render operations:

```
array filtering
map
simple calculations
JSX creation
```

These are just JavaScript operations.

Modern JS engines handle this extremely fast.

Even thousands of renders per second are often fine.

# Example Render Flow in Your Case

Typing in the input:

```
User types "A"
↓
setInputQuery("A")
↓
App() runs again
↓
filteredData recalculates
↓
React compares Virtual DOM
↓
Only necessary DOM updates occur
```

---

# When Re-renders Actually Become Expensive

Re-renders become costly when you have things like:

```
very large lists (10k+ items)
heavy computations
complex charts
large nested component trees
```

In those cases we use optimizations like:

```
React.memo
useMemo
useCallback
list virtualization
```

But these are **optimizations**, not the default requirement.

---

# React's Design Philosophy

React intentionally favors:

```
simple mental model
over premature optimization
```

The mental model is:

```
State changes
↓
component re-runs
↓
React efficiently updates DOM
```

---

# Quick Interview Answer

If asked:

"Does React re-render the whole component on state change?"

Correct answer:

```
Yes, the component function re-executes.

But React only updates the parts of the DOM that actually changed
thanks to Virtual DOM diffing.
```

---

# One More Useful Detail

Even though the component re-runs:

```
DOM nodes are NOT recreated unless necessary.
```

React tries to **reuse existing DOM elements** whenever possible.

---

# Simple Rule to Remember

```
State change → component function runs again
DOM updates → only where differences exist
```

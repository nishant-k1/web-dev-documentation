# dev mode vs prod mode

**In production:**

- Bundler creates optimized chunks
- Files are minified
- Hash names added
- Multiple modules may be combined into fewer optimized chunks

**In dev:**

- Modules are served individually
- No minification
- No heavy chunk optimization
- Mostly separate ES module requests

So splitting still happens — just without aggressive optimization.

## What actually happens with `<script>` and `<link>` in dev vs prod

`In dev mode files are served from your source structure.`
`In production, files are served from the generated build/dist folder.`

If your HTML has:

```html
<link rel="stylesheet" href="/style.css" />
<script type="module" src="/main.js"></script>
```

The browser:

1. Parses HTML
2. Sees <link> → sends HTTP request for style.css
3. Sees <script> → sends HTTP request for main.js
4. Downloads them
5. Parses and executes JS

That behavior is the same in dev and production.
The difference is what server responds and what files look like.

### 🟢 2️⃣ In Development Mode (Vite / Webpack Dev Server)

When you run: npm run dev, You usually get something like: `http://localhost:5173`

Vite or Webpack Dev Server is running.
But instead of creating a big bundle, it:

1. Serves modules individually
2. Uses native ES modules (especially in Vite)
3. Performs on-demand transformation
4. Enables HMR (Hot Module Replacement)

So:

✔ Bundler tool is involved
❌ Traditional bundling into one big file does NOT happen
✔ Module transformation still happens (JSX → JS, TS → JS)
✔ code splitting and lazy loading happens
❌ Tree Shaking, minification, compression doesn't happen

Now:

1. The browser makes HTTP requests
2. The server is your local dev server
3. Files are often served unminified
4. Modules may be served separately (ES modules)
5. No heavy optimization

Example in Dev (Vite):
Instead of one big bundle, the browser may request:

```txt
/src/main.tsx
/src/App.tsx
/node_modules/react/index.js
```

Many small module requests.

So in dev:

- More requests
- No minification
- Faster rebuild, not optimized output

### 🟢 3️⃣ In Production Mode

When you run: npm run build

Now bundler:

- Builds full dependency graph
- BBundles modules into chunks
- BApplies tree shaking
- BMinifies
- BGenerates hashed filenames
- BOptimizes for performance

Now the app is served from:
dist/

You get optimized output like:

```txt
dist/
  index.html
  assets/main.abc123.js
  assets/vendor.xyz456.js
```

Now in production:

1. Browser requests fewer files
2. Files are minified
3. Files are bundled
4. Files may be compressed (gzip/brotli)
5. Often served via CDN

So in prod:

1. Fewer requests
2. Smaller files
3. Faster load
4. Optimized for users

## 🟢 Key Difference Summary dev vs prod

| Aspect       | Dev Mode              | Production Mode  |
| ------------ | --------------------- | ---------------- |
| Server       | Local dev server      | Real server/CDN  |
| JS files     | Many separate modules | Bundled chunks   |
| Minified     | No                    | Yes              |
| Tree shaking | No                    | Yes              |
| Purpose      | Developer experience  | User performance |

🟢 Important Insight

Even in dev mode:

The browser still makes HTTP requests.
The only difference is:

- Who responds (local server vs deployed server)
- How optimized the files are

# critical CSS

Critical CSS = The minimum CSS required to render the visible portion of the page (above the fold).

Above the fold means:

- What the user sees immediately
- Without scrolling

Example:

- Layout
- Hero section styling
- Navigation bar styling
- Fonts for visible text

Not things like:

- Footer styles
- Modal styles
- Hidden dropdown styles
- Page 2 components

## Why critical CSS Placed in `<head>`

- CSS in <head> is discovered early
- It can be parsed before rendering
- It avoids layout shifts

Example:

```html
<head>
  <style>
    body {
      margin: 0;
    }
    .navbar {
      background: black;
    }
    .hero {
      font-size: 24px;
    }
  </style>
</head>
```

This inline CSS helps render first screen faster.
That is called critical CSS if it contains only the necessary styles.

🟢 Important Distinction

Inline CSS ≠ Automatically Critical CSS

If you inline:

```html
<style>
  /* 5000 lines of full app CSS */
</style>
```

That is NOT good critical CSS strategy.

Critical CSS is:

- Small
- Minimal
- Carefully extracted
- Focused on first paint

## How Modern Apps Do This

Frameworks like:

- Next.js
- Astro
- Gatsby

Often automatically extract critical CSS during build.

React (CRA/Vite) by default does not automatically extract critical CSS unless configured.

👉 Yes, during build, the framework analyzes your CSS and may inline the critical part directly into the generated HTML.
👉 Even if you originally wrote it in external CSS files.

Frameworks like:

- Next.js
- Astro
- Gatsby

Analyze:

What HTML is generated
What CSS is actually used on first render
What appears above the fold
Then they split CSS into:

1️⃣ Critical CSS
2️⃣ Remaining CSS

## 🟢 Why This Works Well in SSR Frameworks

Frameworks like Next.js render HTML on the server.

That means they know exactly:

- What components are rendered initially
- What CSS those components use

Because the HTML is generated on the server, the framework can determine which styles are required for the first render.

This makes **critical CSS extraction** much easier and more reliable.

---

## 🟢 Why It’s Harder in Client-Side React (CRA / Vite)

In plain client-side React apps:

- Initial HTML is mostly empty
  ```html
  <div id="root"></div>
  ```
- Real content is rendered later via JavaScript
- CSS usage isn’t known until runtime

Because of this:

Automatic critical CSS extraction is less common  
Unless additional tooling or configuration is used

---

## 🟢 Important Insight

Critical CSS extraction:

- ✔ Happens at build time
- ✔ Modifies generated HTML
- ✔ Improves First Paint
- ✔ Is common in SSR / SSG frameworks

It does **not** modify your source files.

Only the final build output is optimized.

---

## 🟢 Interview-Level Summary

You can say:

> SSR frameworks can extract critical CSS more effectively because they know which components render on the first request. In client-side React apps, this is harder since the initial HTML is mostly empty and content is rendered dynamically.

That shows architectural understanding.

# Middlewares

- Middleware is a function that has access to the request object, response object, and the next function in the application’s request-response cycle. It executes before the final route handler and can modify the request, response, or end the request cycle.
- before the route handler and can also modify the response before it is sent.
- You do not need separate middleware for request and response unless you want separation for clarity.
- Axios interceptors on the client intercept outgoing requests and incoming responses. Backend middleware does the same on the server side within the request–response lifecycle. The difference is that interceptors operate per client request, while middleware is part of the server’s core architecture and affects all incoming requests.
- Middleware helps maintain separation of concerns by extracting cross-cutting logic like authentication, logging, and validation from route handlers.
- Middleware functions execute before the final route handler. They can modify request/response objects, perform authentication, logging, or validation.
- Middleware in Express is just a function with the signature (req, res, next). When we pass that function into app.use() or router.use(), we are registering it in the request-response cycle.
- Signature:

  ```js
  (req, res, next) => {};
  ```

- Flow:

  ```text
  Client → Middleware → Controller → Middleware → Response → Client
  ```

## How Middleware Flow Works (Very Important)

1. Request comes to server
2. Express checks middleware in order
3. Each middleware must call next() to move forward
4. If next() is not called, request will hang
5. Final route handler sends response
6. Order matters. Execution is sequential.

example

```js
app.use((req, res, next) => {
  console.log("Middleware 1");
  next();
});

app.use((req, res, next) => {
  console.log("Middleware 2");
  next();
});

app.get("/", (req, res) => {
  res.send("Final Handler");
});
```

Execution order:

```text
Middleware 1 → Middleware 2 → Route handler
```

## Real Practical Use Cases

- Authentication (JWT verification)
- Logging
- Request validation
- Rate limiting
- CORS handling
- Parsing JSON
- Centralized error handling

## What happens if you don’t call next()?

The request will hang because Express won’t move to the next middleware or route handler.

## Can middleware end the request cycle?

Yes. If it sends a response using res.send(), res.json(), or res.status(), it ends the cycle.

## How to apply middleware only to specific routes?

```ts
app.get("/dashboard", authMiddleware, handler);
```

or

```ts
router.use(authMiddleware);
```

## Middleware identification

```js
app.get("/users", (req, res) => {
  res.send("Users");
});
```

Above, is this callback a middleware

```js
(req, res) => {
  res.send("Users");
};
```

Not that is technically a route handler, not a typical middleware.

**What makes something middleware in Express?**
Middleware must follow this signature:`(req, res, next)`
It participates in the middleware chain and usually calls next().

This function:
`(req, res) => { ... }`
is a route handler.

This is a route handler.

It is still technically middleware internally in Express.

But conceptually, we call it a route handler because:

- It does not call next()
- It ends the request-response cycle, rather than passing to next middleware or route handler
- It contains the main business logic

## Is a route handler middleware?

In Express, route handlers are technically a type of middleware because they follow the same request-response pattern. However, conceptually, middleware usually performs intermediate processing and calls next(), whereas route handlers typically terminate the cycle by sending a response.

## If first one doesn’t call next(), second one will never execute.

If middleware sends a response AND still calls next(), the control will pass to the next middleware.
That’s actually the problem.

What will happen?

1. First middleware sends response.
2. next() is called.
3. Express moves to next middleware.
4. Second middleware tries to send another response.
5. Error occurs:

When you call: `res.send()`, the response is finished.
After that:

1. You should NOT call next()
2. You should NOT try to send another response

## The Middleware Real Rule

A middleware should either:

Call next()
OR
Send a response

But not both.

## If a middleware sends a response but does NOT call next(), is that valid or a mistake? Why?

Yes — valid.

Why Is It Valid?

Because middleware has two responsibilities:

1. Pass control forward using next()
2. Or terminate the request-response cycle by sending a response

If it sends a response, Express considers the request handled.

No need to call next().

## Third-party middleware

Third-party middleware is simply middleware written by other developers and published as npm packages. Instead of writing common logic yourself, you plug in tested libraries.

It is external middleware installed via npm and mounted using:
`app.use(packageName())`

They handle cross-cutting concerns like:

- Security
- Logging
- CORS
- Rate limiting
- File uploads
- Compression

### 1. CORS Middleware

Package: cors
Browsers block cross-origin requests by default. CORS middleware adds proper response headers to allow controlled cross-origin access.

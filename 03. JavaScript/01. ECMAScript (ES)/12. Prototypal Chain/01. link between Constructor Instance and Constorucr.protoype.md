# link between Constructor Instance and Constructor.prototype

- It is starting part of the prototypal chain
- The prototype chain is the full chain created because of that link.
- It is the instance’s internal [[Prototype]] (often accessed as **proto**) that links the instance to Constructor.prototype.

```js
function Person(name) {
  this.name = name;
}

const p1 = new Person("Nishant");
```

When new Person() runs, JavaScript does:

```js
p1.__proto__ = Person.prototype;
```

So the chain becomes:

```text
p1
  ↓
Person.prototype
  ↓
Object.prototype
  ↓
null
```

## What Is the Prototype Chain

- The prototype chain is the lookup path JavaScript follows when you access a property:
- The prototype chain is simply: A linked list of objects connected via [[Prototype]].
- It continues until it reaches null.
- There is no fixed limit like “4 levels”.

```js
p1.toString();
```

JavaScript checks:

- Does p1 have toString? ❌
- Does Person.prototype have it? ❌
- Does Object.prototype have it? ✅
- Return it.

That upward search path is the prototype chain.

**The instance’s internal [[Prototype]] is set to Constructor.prototype. This linkage forms the beginning of the prototype chain.**

Prototype Chain actually is the combined link from instance to Constructor.prototype to Object.prototype to null

```text
instance → Constructor.prototype → Object.prototype → null
```

The constructor function itself is NOT in that chain. The constructor lives on a separate chain:

```text
Person → Function.prototype → Object.prototype → null
```

These are two different chains.

When new is used, the instance’s internal [[Prototype]] is set to the constructor’s .prototype. That linkage creates the start of prototype chain used for inheritance and property lookup.

## the link is parallel or series

If we create multiple instances of Person, are they linked in parallel or in series?

```js
function Person(name) {
  this.name = name;
}

const p1 = new Person("Nishant");
const p2 = new Person("Rahul");
const p3 = new Person("Amit");
```

Now what does the structure look like?
The Structure

```text
p1 ──┐
     │
p2 ──┼──→ Person.prototype → Object.prototype → null
     │
p3 ──┘
```

They are linked in parallel, not in series. Each instance directly points to: `Person.prototype`. They do NOT point to each other.

Each instance gets its own direct link to the Constructor.prototype

**Why This Is Important**
Because methods are shared:

```js
Person.prototype.greet = function () {
  console.log("Hello " + this.name);
};
```

All instances use the same single greet function in memory. That’s the power of prototype-based sharing.

**Clean Mental Model**

Think of it like this:

- Person.prototype is one shared toolbox.
- Every instance has a direct wire connected to that toolbox.
- They do not pass messages through each other.

Multiple instances are linked in parallel to the same Constructor.prototype. They do not form a chain among themselves.

**is this connection called prototypal chain?**
It is one step.

❌ The direct connection itself is not the prototypal chain.
✅ That connection is the start of the prototype chain.

The prototype chain is the entire upward path used during property lookup.

For p1, the full chain is:

```text
p1
  ↓
Person.prototype
  ↓
Object.prototype
  ↓
null
```

That entire structure is the prototype chain.

Think of it like this:

One wire connecting p1 → Person.prototype
👉 That is a prototype link.

The whole staircase going upward
👉 That is the prototype chain.

Why It’s Called a “Chain”. Because JavaScript keeps walking upward when searching for properties:

```js
p1.toString();
```

JS checks:

1. p1
2. Person.prototype
3. Object.prototype
4. null → stop

That upward traversal is what we call:
The prototype chain lookup mechanism.

The instance’s internal [[Prototype]] points to Constructor.prototype. That linkage forms the beginning of the prototype chain, which continues upward until null.

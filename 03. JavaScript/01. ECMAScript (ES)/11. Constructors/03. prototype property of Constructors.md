# prototype

The constructor function creates the object, and `prototype` property of the constructor provides the shared blueprint (methods and inherited properties) for all instances.

When you create a function:

```js
function Person() {}
```

JavaScript implicitly creates a `prototype` property for `Person Constructor object` which is always an `object`.
More specifically:

It’s a plain object
Internally, it starts like:

```js
Person.prototype = {
  constructor: Person,
};
```

Yes — JavaScript allows you to overwrite it:
But this is where things break conceptually.

```js
Person.prototype = 42; // technically allowed
```

```js
Person.prototype;
console.log(typeof Person.prototype); // "object"
```

⚠️ Why it should always be an object
The prototype is used when you create instances:

```js
const p = new Person();
```

JavaScript does:

```js
p.__proto__ === Person.prototype;
```

So if you do this:

```js
Person.prototype = 42;
const p = new Person();
```

Then:

```js
p.__proto__ becomes 42 ❌ (invalid prototype chain)
```

Many JS features stop working correctly

In practice, prototype must be an object or null
⚠️ Important nuance
If you replace the whole prototype:

```js
Person.prototype = {};
```

You lose the default constructor:

```js
console.log(Person.prototype.constructor); // ❌ not Person anymore
```

So better:

```js
Person.prototype = {
  constructor: Person,
  greet() {},
};
```

## `[[Prototype]]` vs `obj.__proto__` vs `Object.getPrototypeOf(obj)` vs `Object.setPrototypeOf(obj, newProto);`vs `Object.create(proto)` vs `prototype chain`

- `__proto__`is a getter/setter that exposes `[[Prototype]]`
- `[[Prototype]]` is an internal slot that every JavaScript object has.
- `[[Prototype]]` is part of the JavaScript engine’s internal mechanics
- You cannot access `[[Prototype]]` directly in standard code
- So in practice:

  ```js
  obj.__proto__ === obj[[Prototype]]; // conceptually true
  ```

But:

- `[[Prototype]]` → internal (real thing)
- `__proto__` → legacy accessor (not recommended in modern code)

- Instead of `__proto__`, use:
  - `Object.getPrototypeOf(obj)`;
  - `Object.setPrototypeOf(obj, newProto)`;

- Example

```js
function Person() {}
const p = new Person();
```

Now internally/implicitly:

```js
p.[[Prototype]] === Person.prototype // ✅ true
```

Using real JS:

```js
Object.getPrototypeOf(p) === Person.prototype; // ✅ true
```

When you access a property over instance object, eg: `p.someMethod()`:
JS does:

1. Look on p
2. If not found → go to `p.[[Prototype]]`
3. Continue up the chain

👉 That chain is called the **`prototype chain`**

| Term                                    | Meaning                                   |
| --------------------------------------- | ----------------------------------------- |
| `[[Prototype]]`                         | Internal hidden link every object has     |
| `__proto__`                             | Old way to access that link               |
| `Object.getPrototypeOf()`               | Correct modern way to access that link    |
| `Object.setPrototypeOf(obj, newProto);` | Object used to set that link during `new` |

### `Object.setPrototypeOf(p, Person.prototype)

👉 It manually changes the internal link ([[Prototype]]) of p.

So after this:

```js
Object.getPrototypeOf(p) === Person.prototype; // ✅ true
```

But wait… doesn’t `new` already do this?
Yes 👍
`const p = new Person();` Internally does: `p.__proto__ = Person.prototype`
So normally, you never need to call setPrototypeOf.
🤔 Then when would you use it?
Case 1: You created a plain object
`const p = { name: "Nishant" };`
Right now:
`Object.getPrototypeOf(p) === Object.prototype`
👉 It has NO connection to Person
Now you attach it manually:
`Object.setPrototypeOf(p, Person.prototype);`

⚠️ Important warning
`Object.setPrototypeOf` is slow and generally discouraged in real-world code.

Why?

- It changes object structure at runtime
- Breaks JS engine optimizations

✅ Preferred approach instead
Use:

```js
const p = Object.create(Person.prototype);
p.name = "Nishant";
```

👉 This sets the prototype correctly from the start

**🧠 Clean mental model**

new Person() → automatic linking
Object.create(proto) → manual but clean
Object.setPrototypeOf(obj, proto) → force change later (rare use)

## prototype chain

```text
p1
  ↓
Person.prototype
  ↓
Object.prototype
  ↓
null
```

# No. of steps in prototype chain

## If I create instances from built-in constructors (like Array, Promise) or from my own constructor function — will they always have the same fixed number of steps in the prototype chain?

- Yes — when created directly from a constructor using new, the structure is usually consistent and predictable.
- But it is not a universal fixed rule. It depends on inheritance.

### Custom Constructor Example

```js
function Person() {}
const p1 = new Person();
```

Prototype chain:

```text
p1
  ↓
Person.prototype
  ↓
Object.prototype
  ↓
null
```

That’s 4 steps (including null).
This is the standard pattern for normal constructor functions.

### Built-in Constructor Example – Array

```js
const arr = new Array();
```

Prototype chain:

```text
arr
  ↓
Array.prototype
  ↓
Object.prototype
  ↓
null
```

Same number of steps.

### Built-in Constructor Example – Promise

```js
const pr = new Promise(() => {});
```

Prototype chain:

```text
pr
  ↓
Promise.prototype
  ↓
Object.prototype
  ↓
null
```

Again — same structure.

### So What’s the Pattern? When you directly create an instance from a constructor (custom or built-in):

```text
instance
  ↓
Constructor.prototype
  ↓
Object.prototype
  ↓
null
```

This is the default shape.

### But Here’s Where It Changes

If inheritance is involved:

```js
class Animal {}
class Dog extends Animal {}

const d = new Dog();
```

Now Prototype chain becomes:

```text
  d
  ↓
Dog.prototype
  ↓
Animal.prototype
  ↓
Object.prototype
  ↓
null
```

Now it’s longer.

## 🔥 The Universal Pattern: Default Case (Direct Constructor)

Length: 4 steps (including null)

Applies to:

Custom constructors
Array
Promise
Date
Map
etc.

```js
new Constructor();
```

You always get:

```text
instance
  ↓
Constructor.prototype
  ↓
Object.prototype
  ↓
null
```

Unless you introduce inheritance or modify prototypes manually.

## There are always two chains:

1. **Instance chain**
   Controls property lookup.

2. **Constructor chain**
   - Because constructors are functions.
   - Keep them separate in your mind.

## The Rule That Never Fails

The prototype chain:

- Starts at the object
- Follows [[Prototype]]
- Ends at null
- Can be any length
- Is determined by inheritance

## All situations where the prototype chain can become longer or different

### ✅ Case 1 — Classical Inheritance (extends / Object.create)

If inheritance is introduced:

```js
class Animal {}
class Dog extends Animal {}

const d = new Dog();
```

Chain becomes:

```text
  d
  ↓
Dog.prototype
  ↓
Animal.prototype
  ↓
Object.prototype
  ↓
null
```

Longer because one prototype inherits from another.
Same thing happens with constructor functions:

```text
Dog.prototype = Object.create(Animal.prototype);
```

### ✅ Case 2 — Object Created From Another Object

```js
const obj1 = { a: 1 };
const obj2 = Object.create(obj1);
const obj3 = Object.create(obj2);
```

Chain:

```text
obj3
  ↓
obj2
  ↓
obj1
  ↓
Object.prototype
  ↓
null
```

Each Object.create() adds one more level. This is pure prototype chaining (no constructor involved).

### ✅ Case 3 — Built-in Types With Internal Inheritance

```js
class MyArray extends Array {}
const arr = new MyArray();
```

Chain:

```text
arr
  ↓
MyArray.prototype
  ↓
Array.prototype
  ↓
Object.prototype
  ↓
null
```

Again, inheritance makes it longer.

### ✅ Case 4 — Custom Prototype Reassignment

```js
function Person() {}

Person.prototype = {
  greet() {},
};
```

Still:

```text
instance
  ↓
NewPrototypeObject
  ↓
Object.prototype
  ↓
null
```

But if you assign:

```js
Person.prototype = Object.create(SomeOther.prototype);
```

Then chain becomes longer depending on SomeOther.

## Case 5 — Shorter Chain (No Object.prototype)

```js
const obj = Object.create(null);
```

Chain:

```text
obj
  ↓
null
```

Very short — no Object.prototype.
Used for dictionary-like objects.

## Important: You cannot create an instance from another instance using new

```js
const arr1 = new Array();
const arr2 = new arr1(); // ❌ ERROR
```

This fails because:

- arr1 is an object
- It is NOT a constructor
- Only functions can be used with new

So instance → instance creation using new is not possible.

✅ What You Can Do: Object.create(instance)
Now this is valid:

```js
const arr1 = [];
const arr2 = Object.create(arr1);
```

Now what happens?
Prototype Chain Becomes Longer
Let’s inspect the chain:

```text
arr2
  ↓
arr1
  ↓
Array.prototype
  ↓
Object.prototype
  ↓
null
```

Yes — now it’s longer.

Because:

```js
arr2.__proto__ === arr1; // true
```

So here:

- arr1 is acting as the prototype of arr2
- arr1 is still an Array instance
- So arr2 inherits from a specific instance

🧠 Important Concept

Normally:

```text
instance → Constructor.prototype → Object.prototype → null
```

But with Object.create(instance):

```text
newInstance → oldInstance → Constructor.prototype → Object.prototype → null
```

Each Object.create() adds one more layer.

⚠ But This Is Rare In Practice

Creating instances from instances is:

- Technically valid
- Rare in real-world code
- Usually avoided because it can cause unexpected behavior

Example:

```js
arr1.push(1);
console.log(arr2[0]); // 1 (inherited!)
```

Because arr2 inherits from arr1.

**If you create an object from another instance using Object.create(instance), the prototype chain becomes longer because the instance itself becomes part of the chain.**
You cannot create an instance from another instance using new.

## What is common in real-world practice that increases prototype chain length?

✅ 1️⃣ Class Inheritance (extends) — Very Common
This is the most common reason the chain becomes longer.

```js
class Animal {}
class Dog extends Animal {}

const d = new Dog();
```

Chain:

```text
  d
  ↓
Dog.prototype
  ↓
Animal.prototype
  ↓
Object.prototype
  ↓
null
```

Every extends adds one more prototype level.
This is extremely common in:

- React component libraries
- Custom Error classes
- Framework internals
- Backend Node apps

✅ 2️⃣ Extending Built-in Types — Also Common
Example:

```js
class MyArray extends Array {}
const arr = new MyArray();
```

Chain:

```text
arr
  ↓
MyArray.prototype
  ↓
Array.prototype
  ↓
Object.prototype
  ↓
null
```

Libraries sometimes extend:

- Error
- Array
- Map
- HTMLElement

So this increases chain depth.

✅ 3️⃣ Custom Error Classes (Very Common in Real Apps)

```js
class AppError extends Error {}
class ValidationError extends AppError {}

const err = new ValidationError();
```

Chain:

```text
err
  ↓
ValidationError.prototype
  ↓
AppError.prototype
  ↓
Error.prototype
  ↓
Object.prototype
  ↓
null
```

This is extremely common in Node.js backends and larger apps.

✅ 4️⃣ Framework-Level Inheritance (You Don’t See It, But It Happens)

Example:
In React (older versions):

```js
class MyComponent extends React.Component {}
```

That creates:

```text
instance
  ↓
MyComponent.prototype
  ↓
React.Component.prototype
  ↓
Object.prototype
  ↓
null
```

Even if you don’t manually think about it, frameworks use inheritance internally.

🚫 What Is NOT Common
These are rare:

- Object.create(instance)
- Manual prototype mutation
- Deep manual prototype chaining
- Reassigning prototype dynamically

Those are mostly theoretical or library-internal use cases.

## 🎯 The Practical Rule

In real-world development, the prototype chain grows longer primarily because of:

- class A extends B
- Built-in class extension
- Error subclassing
- Framework inheritance patterns

Not because of instance-to-instance linking.

🧠 Important Perspective
Even when the chain gets longer, it’s usually:

- 3–6 levels deep
- Not 20 levels deep
- Rarely performance-problematic

JavaScript engines are highly optimized for prototype lookup.

# Code Splitting & lazy loading

## “React.lazy is used for rendering components that are loaded asynchronously. It takes a callback that returns a dynamic import. The dynamic import enables code splitting at build time and downloads the chunk when executed at runtime. React.lazy integrates that async loading with Suspense.”

## Dynamic import

- **Dynamic import returns a Promise that resolves to the module object, where the default export contains the component.**
- When you write: `import("./Dashboard")`, that is called a dynamic import.
- It returns a promise
- So this: `const result = import("./Dashboard");`, result is: `Promise<Module>`, Not the component directly.
- This promise resolves to a module object.
- Example

  If Dashboard.tsx looks like this:

  ```js
  export default function Dashboard() {
    return <h1>Dashboard</h1>;
  }

  export const version = "1.0";
  ```

  Then:

  ```js
  import("./Dashboard").then((module) => {
    console.log(module);
  });
  ```

  The module will look like:

  ```js
  {
  default: DashboardFunction,
  version: "1.0"
  }
  ```

  So dynamic import gives you the full module object.

- Why is it a Promise:
  - The browser may need to download the JS chunk.
  - Network operations are asynchronous.
  - So it must return a Promise.
  - Even if the chunk is cached, it still returns a Promise (but it resolves immediately).

- How React.lazy Uses This
  - React.lazy expects, a function that returns a promise, which is `() => Promise<{ default: Component }>`
    🔹 Important Detail
    This will NOT work: `const Dashboard = await import("./Dashboard");`
    Because then Dashboard becomes the module object, not the component.
    You would need:

    ```jsx
    const module = await import("./Dashboard");
    const Dashboard = module.default;
    ```

    React.lazy handles this automatically.

    🔹 Very Clear Mental Model

    ```text
    import("./Dashboard")
            ↓
    returns Promise
            ↓
    Promise resolves to module object
            ↓
    module.default is the component
    ```

## Code Splitting

### What it is

Code splitting means breaking your application into smaller chunks that load only when needed.

Instead of loading one large JavaScript file, we load:

- Main bundle
- Feature bundles
- Route bundles

### What triggers code splitting

Code splitting is done by: `import('./somePath')`

Specifically:

- The dynamic import syntax
- And the bundler (Webpack/Vite/etc.)

When the bundler sees: `import('./somePath')`, It creates a separate chunk.

Code splitting happens because of dynamic `import()`, not `React.lazy`.

### Why it matters

It reduces:

- Initial load time
- Time to Interactive (TTI)
- First Contentful Paint (FCP)

---

### In React

#### Route-based Splitting

```js
const About = React.lazy(() => import("./About"));
```

---

### In Next.js

Automatic route-level splitting happens by default.

Each file inside the `pages` or `app` directory becomes its own chunk.

---

### How to Explain in Interview

> “I implement route-based and component-level code splitting using React.lazy and dynamic imports to reduce initial bundle size and improve performance.”

Clean. Technical. Confident.

## Lazy Loading

### What it is

Lazy loading means loading resources only when they are needed.

It can apply to:

- Components
- Images
- Routes
- APIs

### What `React.lazy` Actually Does

React.lazy:

- Accepts a function returning a Promise
- Calls that function when the component is rendered
- Suspends rendering until the Promise resolves
- Works with `<Suspense />`

It integrates async loading into React’s rendering system.
It does NOT create chunks.

### React Lazy Example

```js
const Dashboard = React.lazy(() => import('./Dashboard'))

<Suspense fallback={<Loader />}>
  <Dashboard />
</Suspense>
```

---

### Image Lazy Loading

```html
<img src="image.jpg" loading="lazy" />
```

---

## Difference Between Code Splitting & Lazy Loading

| Code Splitting          | Lazy Loading              |
| ----------------------- | ------------------------- |
| Breaks code into chunks | Controls when chunks load |
| Build-time strategy     | Runtime strategy          |

## 🧠 Clean Responsibility Separation

| Responsibility       | Who Handles It               |
| -------------------- | ---------------------------- |
| Code splitting       | Dynamic `import()` + bundler |
| Chunk creation       | Bundler                      |
| Chunk downloading    | `import()` at runtime        |
| Suspense integration | `React.lazy`                 |
| Async rendering      | `React.lazy`                 |

---

## Clean Interview Line

> “Code splitting creates chunks; lazy loading controls when those chunks are loaded.”

# Important Clarification (Very Interview-Ready)

## Are We Using React.lazy for Both?

Short answer: Yes — but they are not the same concept.

This is where many candidates get confused.

---

## 🔹 What `React.lazy` Actually Does

When you write:

```js
const Dashboard = React.lazy(() => import("./Dashboard"));
```

Three things happen:

1. `import()` enables **code splitting** (creates a separate chunk at build time)
2. `React.lazy()` ensures that chunk loads only when rendered (lazy behavior)
3. `Suspense` handles the loading state

So:

- `import()` → Enables code splitting
- `React.lazy()` → Makes the split chunk load lazily
- `Suspense` → Shows fallback while loading

---

## 🔹 Code Splitting vs Lazy Loading (Crystal Clear)

## Code Splitting

Build-time optimization.

The bundler (Webpack / Vite) sees:

```js
import("./Dashboard");
```

And creates a separate file like:

```
dashboard.chunk.js
```

That’s code splitting.

---

## Lazy Loading

Runtime behavior.

The chunk downloads only when:

- The component renders
- The route is visited

That’s lazy loading.

---

## 🔹 Why We Use Suspense

`React.lazy` components are asynchronous.

So React needs:

```jsx
<Suspense fallback={<Loader />}>
  <Dashboard />
</Suspense>
```

Without Suspense → React throws an error.

Suspense:

- Waits for component
- Shows fallback UI
- Renders component once loaded

---

## 🔹 Very Important Interview Clarity

If interviewer asks:

**“Are code splitting and lazy loading the same?”**

You say:

> “Not exactly. Code splitting is the technique of breaking bundles into smaller chunks. Lazy loading is the strategy of loading those chunks only when needed. React.lazy with dynamic import enables both.”

That sounds senior and precise.

---

## 🔹 They Are Not Always the Same

You can do code splitting without `React.lazy`.

Example:

```js
button.onclick = () => {
  import('./heavyChart').then(...)
}
```

Still code splitting.

You can also lazy load:

- Images (`loading="lazy"`)
- Data (fetch only when user clicks)
- Routes (Next.js automatic splitting)

So lazy loading is broader than `React.lazy`.

---

## 🔹 Real-World Flow

When user visits homepage:

- Only main bundle loads

When user clicks Dashboard:

- Browser downloads `dashboard.chunk.js`
- Suspense shows loader
- Component renders

This improves:

- Initial load
- LCP
- TTI

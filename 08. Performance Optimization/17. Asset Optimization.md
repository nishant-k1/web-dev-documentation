# 4️⃣ Asset Optimization

```text
React → renders components
Browser → loads fonts
Preload → increases font priority
font-display: swap → prevents invisible text
```

---

Focus: Optimize static resources so that pages load faster and layout stays stable.

---

## What You Optimize

- **Images**
  - Use modern formats (WebP, AVIF)
  - Compress images
  - Use responsive images (`srcset`, `sizes`)
  - Lazy-load below-the-fold images (`loading="lazy"`)

- **Fonts**
  - Preload critical fonts
    In React

    Step 1: Inside your index.html (public folder)
    Eg:

    ```jsx
    <link
      rel="preload"
      href="/fonts/Inter-Variable.woff2"
      as="font"
      type="font/woff2"
      crossorigin
    />
    ```

    - Tells browser: “Download this font immediately.”
    - High priority request
    - Happens before CSS parsing completes
    - Reduces FOUT/FOIT

    Step 2: Then Define the Font in CSS

    ```css
    @font-face {
      font-family: "Inter";
      src: url("/fonts/Inter-Variable.woff2") format("woff2");
      font-display: swap;
    }
    ```

    Important part:`font-display: swap;`: This prevents invisible text while the font loads.

  - Subset fonts to only required glyphs
  - Avoid blocking font loading (use `font-display: swap`)

  Why Preload Improves Performance

  **Without preload:**

      - Browser loads HTML
      - Parses CSS
      - Finds @font-face
      - Starts downloading font
      - Text may re-render → layout shift

  **With preload:**

      - Browser starts downloading font immediately
      - CSS finds it already loading
      - Faster text rendering
      - Less layout instability

  “Fonts are preloaded using <link rel="preload" as="font"> in the HTML. React itself doesn’t manage font loading. Preloading ensures the browser downloads fonts early, reducing layout shifts and improving perceived performance.”

- **Layout Stability**
  - “Layout stability requires reserving space before content loads.
  - This can be done using HTML width and height attributes or using CSS with fixed height or aspect-ratio.
  - The key is that the browser must know the dimensions before the resource loads.
  - Always define `width` and `height` (or aspect-ratio) for images/media
  - Reserve space for ads or dynamic widgets

  Browsers prioritize HTML width/height for images because:
  - It allows layout calculation before CSS fully loads.
  - It improves CLS score.
    But CSS aspect-ratio is now equally solid in modern browsers.

## FAQs

Q:
if we do below in css that won't work for layout stability?

- Always define `width` and `height` (or aspect-ratio) for images/media
- Reserve space for ads or dynamic widgets

Answer:
Yes, defining size in CSS can work for layout stability — but only if it reserves space before the content loads.

What Causes Layout Shift?

- The browser doesn’t know the size of something (like an image).
- It renders the page.
- Later the image loads.
- The layout reflows.
- Everything jumps.

That jump = layout instability.

1️⃣ Using HTML width & height (Best Practice): `<img src="/banner.jpg" width="800" height="400" />`

This works because:

- The browser calculates the aspect ratio immediately.
- Space is reserved before the image downloads.
- No layout shift.

Modern browsers compute: **aspect-ratio = width / height**

3️⃣ Using CSS aspect-ratio (Modern & Good)
This is the best CSS-based solution:

```css
.image-container {
  aspect-ratio: 16 / 9;
  width: 100%;
}
```

Now:

- Browser knows ratio
- Space is reserved
- Image loads into that box

No layout shift

- This is production-grade and fully acceptable

## Relation to Core Web Vitals

- Better image and font handling improves:
  - **LCP** (Largest Contentful Paint)
  - **CLS** (Cumulative Layout Shift)

You can say:

> “I optimize images (WebP, lazy loading, responsive sizes) and fonts (preload, subset, font-display) to improve LCP and avoid layout shifts, which directly improves Core Web Vitals.”
